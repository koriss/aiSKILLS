# Сценарии политики чата (для тестов и регрессии)

## Сценарий A — короткий фактчек

**Вход:** один тезис, узкий запрос.

**Ожидаемое поведение:**

- Промежуточных сообщений: **0** (или **1** короткое ack — допустимо).
- В чат после работы: **одно** финальное сообщение по `templates/final-chat-response.txt` **после** сохранения и отправки HTML.
- В `execution-log.json`: `chat_intermediate_count` ∈ {0, 1}, `chat_final_text_sent`: true.

**Негатив:** сообщение «запустил проверку» с планом из 5 пунктов.

---

## Сценарий B — длинное расследование (orchestration)

**Вход:** жирная задача, 3–5 subagent-ов, dosbor возможен.

**Ожидаемое поведение:**

- В чат **не** отправлять статусы по subagent, retry, dosbor.
- Промежуточных: **максимум 1** ack + при реальной задержке **максимум 1** сообщение:  
  `Проверка идёт дольше обычного: добираю первичные источники и противоречия.`  
  Итого ≤ **2** промежуточных, чаще **0–1**.
- Финал: HTML на диск → проверка → отправка HTML → **один** финальный текст → стоп.

**Негатив:** серия сообщений «subagent 1 готов», «жду остальных», «результат через минуту».

---

## Сценарий C — ошибка / слабый subagent

**Вход:** subagent вернул слабый результат, сработали retry или dosbor.

**Ожидаемое поведение:**

- Recovery и критика только в логах и артефактах.
- В чат — **без спама**; пользователь получает **один** финальный ответ с честными оговорками и `Доверие: Low/Medium`, плюс HTML как есть (если создан).
- Если HTML невозможно собрать, финальный текст объясняет границы; не слать второй «итог» теми же словами.

**Негатив:** сырые JSON или выдача поиска в чат; два почти одинаковых финала подряд.
