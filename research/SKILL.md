# Research (OpenClaw): расследования и проверка тезисов

## Overview

Этот skill превращает агента OpenClaw в **исследовательский контур** для команд вида «исследуй», «проверь», «разбери», «это правда или ложь?». Аналитическое ядро — методология **deep-investigation-agent** (факты, гипотезы, карта доказательств, quality gates). Поверх неё — **supervisor-слой**: планирование, разбиение на subagent-ов (3–5), критика результатов, досбор (dosbor), retry/re-split, сохранение артефактов, HTML-отчёт, краткая Telegram-сводка, машиночитаемые логи.

Ключевые принципы:

- ширина источников ≠ глубина; повтор пересказов ≠ подтверждение;
- каждый subagent возвращает **анализ и артефакт**, а не сырой JSON поиска;
- финальный синтез не принимает слабые куски без критики и при необходимости dosbor;
- кириллица и смысл не ломаются; переводы — силами LLM, без внешних переводчиков.

Подробности: `references/orchestration.md`, `references/retry-and-recovery.md`, `references/report-format.md`, `references/telegram-output.md`, `references/logging-and-evaluation.md`.

## When to use

- Запросы на глубокое исследование спорной или быстро меняющейся темы (режим **исследуй**).
- Проверка тезисов из текста, пересказа поста Telegram, цитаты (режим **проверь**).
- Когда важны HTML-отчёт, воспроизводимость (JSON-логи), контроль качества и разбиение сложной задачи на subagent-ов.

## When not to use

- Тривиальный факт без риска и без необходимости верификации.
- Чистое творчество без фактологии.
- Задача явно укладывается в один короткий ответ без многошагового сбора данных.

## Workspace layout (локальная конвенция)

Базовый корень (в контейнере агента по умолчанию):

- `skills/` — каталог skills (этот пакет).
- `prompts/` — сохранённые нормализованные запросы / контракты задач (опционально).
- `reports/` — HTML, subagent `.txt`/`.json`, итоговые JSON.
- `memory/` — краткие заметки о прошлых прогонах, уроки (опционально).

Шаблоны путей:

- HTML: `/home/node/.openclaw/workspace/reports/<slug>-YYYY-MM-DD.html`
- Subagent: `/home/node/.openclaw/workspace/reports/sub<N>-<slug>-YYYY-MM-DD.txt` (доп. `.json`/`.md`/`.log` по необходимости)
- `final-report.json`, `subagent-results.json`, `execution-log.json`, `quality-review.json` — в том же `reports/` рядом с HTML, с тем же `<slug>` и датой.

**Slug:** латиница в `kebab-case`, 8–48 символов, из темы запроса (ASCII); нормализация: нижний регистр, только `[a-z0-9-]`, схлопывание `--`, обрезка по границе слова при превышении длины.

## Режимы

| Режим | Ключевые слова (эвристика) | Фокус |
|-------|---------------------------|--------|
| **investigate** | исследуй, разбери, собери картину, аналитика | Сбор данных, противоречия, гипотезы, записка, уверенность |
| **verify** | проверь, правда ли, фактчек, опровергни | Атомарные утверждения, статусы проверки, первичка |

Если намерение смешанное — по умолчанию **verify**, если явно запрошена «картина»/«сценарии» — **investigate**. Записать выбор и обоснование в `execution-log.json`.

## Runtime workflow

Выполняй по порядку; не пропускай quality gates в конце.

1. **Нормализация запроса**  
   Зафиксируй `request_id`, `chat_id` (если доступен), исходный текст, **нормализованный вопрос**, черновик режима (investigate/verify).

2. **Оценка «жирности» и риска переполнения контекста**  
   Эвристики по умолчанию (см. `references/orchestration.md`):
   - задача «жирная», если: >800 слов входа ИЛИ >8 тезисов/утверждений ИЛИ >6 именованных сущностей ИЛИ широкий временной диапазон (>2 лет) ИЛИ пользователь явно перечисляет несколько независимых подзадач;
   - риск overflow: высокий при множестве подвопросов + длинный контекст канала; оценка «примерный объём» логируется.

3. **Выбор single-agent vs orchestration**  
   - По умолчанию **single-agent**, если задача не жирная и нет признаков конфликтующих объяснений.  
   - Иначе — **orchestration** с **3–5** параллельными subagent-ами (не более **5** одновременно).  
   Запиши в execution log: `why_multiaagent_or_single_agent`, `chosen_strategy`.

4. **Декомпозиция (только для orchestration)**  
   Порядок предпочтения стратегий разбиения: подвопросы → кластеры тезисов → типы доказательств → временные периоды → акторы/стороны → конкурирующие гипотезы.  
   Не резать «по абзацам» без смысловой связи. Каждый кусок: узкая тема, самостоятельный evidence bundle, явный task contract.

5. **Запуск subagent-ов**  
   Каждому subagent-у выдай: цель, границы, минимум поисков (если нужен веб), требование к цитатам/годам, формат выхода, путь файла артефакта.  
   После завершения — **обязательная критическая оценка** (чеклист и рейтинг ⭐–⭐⭐⭐⭐⭐). Формат: `[Имя subagent]: [Оценка] — [почему]`.

6. **Провал и recovery**  
   См. `references/retry-and-recovery.md`: failed → диагностика → при освобождении слота re-split проблемного куска → ограниченное число retry.

7. **Dosbor**  
   Если критика выявила пробелы, противоречия или «библиографию без доказательной базы» — точечные микро-задачи, 1–3 дополнительных subagent-а, без полного дублирования. Максимум **3** раунда dosbor по умолчанию.

8. **Синтез**  
   Объедини результаты с **deep-investigation-agent**: факты, claims, timeline, гипотезы, противоречия, confidence. Заполни блоки HTML и `investigation_report` в `final-report.json`.

9. **Артефакты на диск**  
   Сохрани: HTML (по шаблону `templates/report.html`), `final-report.json`, `subagent-results.json`, `execution-log.json`, `quality-review.json`, subagent-файлы.

10. **Проверка записи**  
    Убедись, что файлы существуют, размер HTML > 0, нет битой кодировки.

11. **Отправка**  
    Сразу после успешного сохранения отправь **HTML-файл** в тот чат, откуда пришёл запрос (механизм OpenClaw: документ/attach по правилам среды). Зафиксируй `send_status` в execution log.

12. **Telegram summary**  
    Одно сообщение, русский язык, стиль аналитической записки, ≤ **95%** лимита одного сообщения (4096 символов → ориентир **3890** символов), см. `references/telegram-output.md`.

13. **Финальный quality review**  
    Заполни `quality-review.json` и пройди **Quality gates**.

## Tool policy

- **Поиск и чтение источников:** используй по максимуму разрешённые инструменты веб-поиска/чтения страниц; для режима verify минимум **5–7** релевантных поисковых шагов на всю задачу (не обязательно на каждый subagent, но при слабом куске — досбор). Один поиск «для галочки» — нарушение.
- **Файлы:** обязательное сохранение артефактов в `reports/`; пути предсказуемые, см. выше.
- **Канал Telegram:** после сохранения HTML — отправка файла в исходный чат; summary текстом отдельным сообщением при необходимости (если платформа разделяет файл и текст).
- **Никакой подмены анализа сырыми выдачами инструментов** в финальном ответе пользователю.

## Memory policy

- Краткие **lessons** (что сработало/сломалось) можно дописывать в `memory/research-lessons.md` (опционально), без PII.
- Не хранить секреты; не дублировать полные логи в memory — только выводы.
- Для повторных запросов по той же теме: проверить `memory/` на предмет прошлых slug/дат, не смешивать разные инциденты.

## Output requirements

- **Язык:** весь пользовательский вывод (HTML, Telegram, пояснения) — **русский**; цитаты на иных языках — с переводом LLM и сохранением смысла.
- **HTML:** самодостаточный файл, секции из `references/report-format.md`; типографика, без сломанной кириллицы.
- **Машиночитаемые JSON:** валидация по `schemas/*.schema.json` скриптом `scripts/validate_artifacts.py`.
- **Совместимость с deep-investigation-agent:** поле `investigation_report` в `final-report.json` соответствует контракту исследования (см. `../deep-investigation-agent/schemas/investigation_report.schema.json`); расширения — в `orchestration_meta`.

## Quality gates

Перед завершением **все** пункты:

1. Режим и стратегия (single/multi) задокументированы в `execution-log.json`.
2. Для orchestration: 3–5 осмысленных subagent-задач или обоснованный меньший split; нет дубликатов исследования между subagent-ами без причины.
3. По каждому subagent есть оценка и рейтинг; мусор не попал в финал как «доказательство».
4. HTML создан по пути шаблона, файлы записаны, размер проверен.
5. `send_status` отражает попытку отправки HTML в чат; ошибка отправки залогирована.
6. Telegram summary одним сообщением, в лимите, без потока ссылок.
7. `quality-review.json` заполнен честно (слабые места указаны).
8. Нет необоснованного статуса «подтверждено» при одной цепочке пересказов.
9. Важные числа и цитаты проверены по контексту (где применимо).

Иерархия источников и правила доказательности — в `references/report-format.md` и в workflow deep-investigation-agent.
