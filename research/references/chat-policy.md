# Политика сообщений в чат (anti-spam)

Цель: **по умолчанию молча работать → один итоговый текст**, без перечисления планов, статусов subagent и «жду результата». Вся техника — в логах.

## Разрешённые сообщения пользователю

| Тип | Когда | Лимит |
|-----|--------|--------|
| Подтверждение приёма (опционально) | Только если без него неочевидно, что задача принята | **Максимум 1** короткое сообщение (1–2 предложения), без плана из 3–6 пунктов |
| Уведомление о задержке | Задача объективно идёт существенно дольше типичного прогона для такого класса запросов | **Ровно 1** допустимое сообщение, фиксированная формулировка (см. ниже). После него — только финал |
| Финальный ответ | После HTML: сохранён, проверен, отправлен | **Ровно 1** сообщение, строгая структура (см. `templates/final-chat-response.txt`) |

Если можно обойтись без подтверждения приёма — **не пиши** промежуточное сообщение вообще (0 служебных до финала).

### Единственная разрешённая формулировка задержки

```
Проверка идёт дольше обычного: добираю первичные источники и противоречия.
```

Иное содержание для «промежуточного» статуса **запрещено**, кроме опционального короткого ack (см. выше).

## Запрещённые шаблоны (не отправлять в чат)

- «Запустил проверку…» + длинный список шагов
- «Жду», «результат через несколько минут», «скоро пришлю»
- Перечень плана на 3–6 пунктов в отдельном сообщении
- Отдельные статусы по каждому subagent
- Пустые сообщения
- Два финальных ответа подряд с дублирующим смыслом
- «Проверка завершена», если следом всё равно идёт полноценный итог (объединить в одно финальное)
- Сырые JSON / сырые search results вместо нормального текста
- Техническая кухня orchestration (retry, re-split, dosbor, quality review) **без явного запроса пользователя**

## Куда писать вместо чата

Все служебные события — только артефакты:

- `execution-log.json` — запуск subagent, retry, timeout, re-split, dosbor-раунды, отправка HTML
- `subagent-results.json`, файлы `sub*.txt` — детали subagent-ов
- `quality-review.json` — самокритика
- При необходимости отдельный `.log` в `reports/`

## Явные условия ошибки поведения (failure)

Считать прогон **нарушающим политику**, если:

1. В чат ушёл «план работ» или перечисление этапов без запроса пользователя.
2. Более **одного** промежуточного сообщения (кроме сочетания: опциональный ack + одно сообщение задержки; итого максимум **2** промежуточных, и то редко).
3. После финального ответа отправлено ещё что-то по той же задаче (кроме системных ошибок доставки).
4. Финальный текст отправлен **до** успешной попытки отправить HTML (если HTML обязателен для этого режима).

Корректное завершение: **HTML → диск → проверка → отправка HTML → один финальный текст → стоп**.
